# Voice Cooking Companion Kit Implementation Plan

## Goal

Build this repo into a complete voice cooking companion kit: a small standalone server, a static recipe-management UI, and the ESP32-S3 hardware files needed to build and flash the countertop device.

The server replaces the Pinata/OpenClaw agent runtime while preserving the useful hardware-facing API contract.

The first implementation should prioritize:

- stable `/query` and `/query-audio` endpoints
- compatibility with the existing ESP32-S3 hardware firmware
- a simpler, more inspectable `/query-audio` architecture
- container hosting on local hardware or remote hosting services
- recipe storage and upload through a simple static frontend at the site root
- hardware docs, firmware, and reference materials in `hardware/`

## Non-Goals

- Do not reuse the Pinata/OpenClaw runtime or workspace architecture.
- Do not reuse the existing Next.js frontend.
- Do not build a general chat agent.
- Do not persist uploaded audio or generated speech as durable user data.
- Do not invent recipes that are not stored locally.
- Do not write personal details, secrets, tokens, private deployment specifics, or other sensitive information into source, docs, tests, logs, commits, or examples.

## Repository Shape

This repo should feel like the full project kit, not only a backend service.

Top-level areas:

- `src/`: standalone server code
- `public/`: static frontend served from `/`
- `hardware/`: ESP32-S3 sketch, hardware README, and reference PDF
- `data/`: local SQLite data, ignored by git
- `generated-audio/`: temporary TTS output, ignored by git
- `docs/`: implementation and architecture notes

## Proposed Stack

- Node.js 20+
- TypeScript
- Fastify HTTP server
- SQLite through `better-sqlite3`
- Static HTML/CSS/JS frontend served by Fastify
- OpenAI HTTP APIs for speech-to-text, optional intent parsing, optional recipe markdown coercion, and text-to-speech
- Docker image for local or remote container deployment

## Token Protection

The old Pinata deployment protects device endpoints with `?token=...`.

This server should preserve that shape:

- if `API_TOKEN` is unset, endpoints are open for local development
- if `API_TOKEN` is set, protected routes require either:
  - `?token=<API_TOKEN>`
  - `x-api-token: <API_TOKEN>`

Protected routes when `API_TOKEN` is set:

- `POST /query`
- `POST /query-audio`
- recipe import/write endpoints
- any endpoint that calls the OpenAI API

Read-only frontend and recipe list endpoints can remain public by default. Recipe upload and OpenAI-backed actions should require the token when `API_TOKEN` is configured.

## API Compatibility

Canonical endpoints:

- `POST /query`
- `POST /query-audio`

OpenClaw relics:

- do not implement `/app/query`
- do not implement `/app/query-audio`
- do not implement `/app/api/audio/:name`

The hardware firmware should be configured to call `/query-audio` directly.

### `/query`

JSON text endpoint.

Supported request modes:

```json
{
  "inputMode": "query",
  "query": "How do I make lemon garlic pasta?",
  "sessionId": "sess_123"
}
```

```json
{
  "inputMode": "next_step",
  "sessionId": "sess_123"
}
```

No audio should be generated by `/query`.

### `/query-audio`

Multipart voice endpoint:

- `Content-Type: multipart/form-data`
- file field `audio`, with `file` accepted as fallback
- optional `sessionId`

JSON next-step endpoint:

```json
{
  "inputMode": "next_step",
  "sessionId": "sess_123"
}
```

For compatibility with the hardware sketch, responses should include:

- `ok`
- `transcript`
- `intent`
- `answerText`
- `audio.url` when TTS succeeds
- `session.id`
- `session.activeRecipeId`
- `session.stepIndex`
- `session.phase`

## `/query-audio` Flow

The implementation should keep the route layer thin.

Flow for multipart audio:

1. authenticate token if configured
2. parse multipart upload
3. resolve or create session id
4. validate audio file type and size
5. transcribe audio to text
6. pass transcript into the same cooking query service used by `/query`
7. update deterministic session state
8. synthesize speech for `answerText`
9. save generated MP3 under `generated-audio`
10. return JSON with a temporary `audio.url`

Flow for JSON next-step:

1. authenticate token if configured
2. parse JSON body
3. resolve session id
4. advance deterministic session state
5. synthesize speech for the response
6. return JSON with a temporary `audio.url`

The core rule: model calls may classify user intent, but deterministic code owns recipe facts and state transitions.

## Query Architecture

Suggested modules:

- `src/server.ts`: Fastify setup
- `src/routes/query.ts`: `/query` and `/query-audio` transport concerns
- `src/routes/recipes.ts`: recipe listing and markdown import endpoints
- `src/services/query-service.ts`: cooking state machine and response shaping
- `src/services/openai.ts`: STT, TTS, intent parsing, markdown coercion
- `src/storage.ts`: SQLite persistence
- `src/markdown/recipe-parser.ts`: local markdown parsing and validation
- `src/auth.ts`: token guard
- `src/audio.ts`: generated audio file writing and serving helpers

## Cooking Session Model

Keep session state narrow:

```ts
{
  activeRecipeId: number | null;
  phase: "ingredients" | "steps";
  index: number;
  pendingPrompt: "ingredients_or_first_step" | "ingredients_or_repeat" | null;
}
```

Public response shape:

```json
{
  "session": {
    "id": "sess_123",
    "activeRecipeId": "12",
    "stepIndex": 0,
    "phase": "ingredients"
  }
}
```

## Recipe Markdown Upload

The static frontend should support uploading one markdown file at a time.

Preferred format:

```md
# Recipe Title

Description text.

## Ingredients

- 1 cup rice
- 2 cups water

## Instructions

1. Rinse the rice.
2. Cook until tender.

## Tags

- pantry
- vegetarian

## Notes

Optional notes.
```

Validation should require:

- one non-empty title
- at least one ingredient
- at least one instruction

Light validation warnings:

- missing description
- very long ingredient lines
- instructions that look like ingredients
- ingredient section found but empty
- instruction section found but empty

Optional LLM coercion:

- if local parsing fails and `OPENAI_API_KEY` is present
- because this path uses the OpenAI API, it must require `API_TOKEN` when token protection is configured
- submit the markdown to a constrained JSON-normalization prompt
- validate the returned JSON before writing to SQLite
- report whether LLM coercion was used

Duplicate title handling:

- if an imported recipe title matches an existing recipe, do not silently overwrite it
- return a response that names the duplicate and asks whether the user wants to update the existing recipe
- the frontend should show that confirmation before sending an explicit update request

Recipe import endpoint:

- `POST /api/recipes/import-markdown`
- protected by token when `API_TOKEN` is set
- accepts multipart file upload or JSON `{ "markdown": "..." }`
- returns parsed recipe, warnings, and saved recipe id

## Generated Audio Pruning

Generated TTS files should be temporary.

Implementation:

- write MP3s under `generated-audio`
- serve by filename from `GET /audio/:name`
- include a script like `npm run prune-audio`
- delete files older than `GENERATED_AUDIO_TTL_HOURS`

Container scheduling options:

- run prune as a host cron job calling `docker exec ... npm run prune-audio`
- run prune with a separate scheduled container
- optionally add an in-process startup prune, but avoid a forever background scheduler in v1 unless needed

## Testing and Verification

Minimum checks:

- TypeScript build
- server starts locally
- `POST /query` rejects invalid mode
- `POST /query` can load seeded recipe
- `POST /query` can advance next step
- `POST /query-audio` rejects missing audio cleanly
- token guard blocks protected endpoints when configured
- markdown import rejects invalid markdown
- markdown import accepts preferred markdown format
- prune script deletes stale MP3s and keeps fresh files

Hardware-facing smoke test:

- configure firmware URL as `/query-audio`
- use `?token=...` when `API_TOKEN` is set
- confirm response includes `session.id`, `answerText`, and `audio.url`
- confirm the returned MP3 URL downloads promptly

## Resolved Decisions

- any endpoint that uses the OpenAI API should require `API_TOKEN` when token protection is configured
- duplicate markdown imports should mention the duplicate name and ask whether to update the recipe
- `audio.url` should be relative by default; the sketch already joins relative audio URLs to `API_ORIGIN`
- deeper `/query-audio` improvements and query event logging should be a fourth PR
- uploading recipes should require a token when `API_TOKEN` is configured, but the whole static page should not be restricted
- copy the full `hardware/` folder into this repo and patch the copied sketch/docs to use `/query-audio`
